# Code Quality

## Principles

- Functions do one thing. If it needs a section comment, extract that section.
- No magic values ‚Äî extract numbers, strings, and config to named constants.
- Handle errors at the boundary. Don't catch and re-throw without adding context.
- No premature abstractions. Three similar lines > a helper used once.
- Don't add features or "improve" things beyond what was asked.
- No dead code or commented-out blocks. Git has history.
- Composition over inheritance.

## Naming (Python / Django)

- **Modules / files**: `snake_case.py` ‚Äî `csv_import.py`, `state_machine.py`.
- **Packages / directories**: `snake_case/` ‚Äî `tcms_requirements/exports/`.
- **Classes**: `PascalCase` ‚Äî `RequirementCategory`, `JiraIntegrationConfig`.
- **Functions / methods / variables**: `snake_case` ‚Äî `get_jira_export_mapping`, `build_json_payload`.
- **Constants**: `SCREAMING_SNAKE_CASE` ‚Äî `DEFAULT_LEVEL_PROFILE`, `REQUIRED_COLUMNS`.
- **Booleans**: `is_`, `has_`, `should_`, `can_` prefix ‚Äî `is_active`, `has_permission`.
- **Private helpers**: leading underscore ‚Äî `_resolve_fks`, `_flatten`.
- **Django model fields**: `snake_case` with FK suffix `_id` used only on the automatic column (`category_id`); the relation is `category`.
- **URL names**: `kebab-case` ‚Äî `requirement-list`, `requirement-export-hub`.
- **Template files**: `snake_case.html` ‚Äî `export_hub.html`, `confirm_delete.html`.

## Django specifics

- Models: class docstring explains the entity's role. Fields grouped by concern (identity / taxonomy / organisation / lifecycle / audit) with blank lines and `# ‚îÄ‚îÄ section ‚îÄ‚îÄ` dividers for readable scrolling.
- ForeignKeys: declare `on_delete` explicitly; prefer `PROTECT` for taxonomy FKs, `SET_NULL` for organisation FKs, `CASCADE` only for through-tables.
- `related_name` is always set explicitly. Never rely on Django's default `_set`.
- Migrations: autogenerated by `makemigrations`, reviewed before commit. Hand-written migrations only when cross-app dependencies force it (see `0001_initial.py`).
- Views: class-based (`LoginRequiredMixin` + `PermissionRequiredMixin`). Avoid function-based views unless there's a strong reason.
- Lazy-import Kiwi core models (`from tcms.testcases.models import TestCase`) inside functions ‚Äî NEVER at module top level. Prevents startup-time import loops and lets the plugin boot even when Kiwi is partially loaded.

## Comments

- **WHY**, never WHAT. If the code needs a "what" comment, rename instead.
- Comment: non-obvious decisions, workarounds with issue links, regex patterns, cross-app FK dependencies, Kiwi gotchas (two-jQuery, middleware ordering).
- Don't comment: obvious code, self-explanatory function names, section dividers, type info.
- No commented-out code ‚Äî delete it. No journal comments ‚Äî git blame does this.
- Docstrings at module boundaries (top of file + class) only; not every internal function.

## Code Markers

| Marker | Use |
|---|---|
| `TODO(author): desc (#issue)` | Planned work |
| `FIXME(author): desc (#issue)` | Known bugs |
| `HACK(author): desc (#issue)` | Ugly workarounds (explain the proper fix) |
| `NOTE: desc` | Non-obvious context for future readers |

Must have an owner + issue link. Don't commit TODOs you can do now. Never use `XXX`, `TEMP`, `REMOVEME`.

## Imports

- **Order**: stdlib ‚Üí third-party (Django, simple_history, requests) ‚Üí internal (`tcms_requirements.*`) ‚Üí local (`from .foo import`). Blank line between groups.
- `# noqa: WPS433` comments are fine on lazy imports inside functions ‚Äî they deliberately violate "imports at top of file" to break startup loops.
- Never import from Kiwi core (`tcms.*`) at module top level ‚Äî always lazy-import inside functions.
